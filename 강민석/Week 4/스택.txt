Java 실무와 코딩 테스트의 함정: java.util.Stack은 쓰면 안 될까?
Java로 스택을 구현할 때 가장 많이 하는 실수 중 하나가 바로 기본 제공되는 Stack 클래스를 사용하는 것이다. 공식 문서에서도 이 클래스의 사용을 지양하고 있다.

상속의 오용: Java의 Stack은 Vector 클래스를 상속받아 만들어졌다. Vector는 인덱스를 통한 중간 삽입/삭제가 가능하므로, LIFO라는 스택의 기본 원칙을 깨고 중간에 있는 데이터를 조작할 수 있는 설계적 결함이 있다.

성능 저하: Vector의 모든 메서드에는 스레드 안전성을 위한 synchronized 키워드가 붙어 있다. 단일 스레드 환경에서도 불필요한 동기화 락(Lock)을 잡기 때문에 성능이 크게 떨어진다.

모범 답안: Java에서 스택이 필요할 때는 Deque (Double-Ended Queue) 인터페이스와 ArrayDeque 구현체를 사용하는 것이 표준이다.

Java

Deque<Integer> stack = new ArrayDeque<>();
stack.push(1); // 스택처럼 사용 가능
구현 방식에 따른 트레이드오프 (Array vs. Linked List)
스택을 직접 구현한다고 가정할 때, 내부적으로 배열을 쓸지 연결 리스트를 쓸지에 따라 특성이 완전히 달라진다.

동적 배열 (Dynamic Array) 기반:
장점: 데이터가 메모리상에 연속적으로 존재하므로 CPU 캐시 지역성(Cache Locality)이 뛰어나 속도가 빠르다.
단점: 할당된 배열의 크기를 초과하면, 더 큰 배열을 새로 할당하고 기존 데이터를 모두 복사하는 Resize 작업이 발생하여 순간적으로 $O(N)$의 시간이 소요된다.

연결 리스트 (Linked List) 기반:
장점: 데이터 추가/삭제 시 Resize 작업이 전혀 발생하지 않아 메모리가 허용하는 한 무한히 늘어날 수 있다.
단점: 각 노드마다 다음 노드를 가리키는 포인터를 저장해야 하므로 메모리 오버헤드가 발생하며, 메모리 파편화로 인해 캐시 효율이 떨어진다.

알고리즘의 꽃: 단조 스택 (Monotonic Stack)
알고리즘 문제에서 스택이 어려운 문제로 출제될 때 십중팔구 사용되는 핵심 기법이다.

개념: 스택 내부의 원소들이 항상 오름차순 혹은 내림차순을 유지하도록 억지로 조작하는 스택이다. 새로운 원소가 들어올 때, 스택의 끝에 있는 원소가 오름/내림차순 조건을 깨뜨린다면 조건을 만족할 때까지 pop을 시켜버린다.

왜 쓸까?: 주로 나보다 큰(혹은 작은) 다음 원소 찾기 류의 문제에 쓰인다. 이중 for문을 돌면 $O(N^2)$이 걸려 시간 초과가 나는 문제를 단조 스택을 활용하면 한 번의 순회로 해결하여 $O(N)$으로 시간 복잡도를 혁신적으로 줄일 수 있다. (예: 백준 오큰수, 히스토그램에서 가장 큰 직사각형 문제)

시스템 아키텍처 관점: Call Stack과 구문 분석
OS와 컴파일러가 스택을 대하는 방식이다.

스택 프레임 (Stack Frame): 함수가 호출될 때마다 메모리의 스택 영역에 함수의 매개변수, 지역 변수, 그리고 함수가 끝나고 돌아갈 반환 주소(Return Address)가 묶인 스택 프레임이 쌓인다. 재귀 함수가 종료 조건을 만나지 못하고 계속 호출되면, 이 스택 영역의 메모리 한계를 초과하여 발생하는 것이 바로 StackOverflowError이다.

컴파일러의 수식 계산: 우리가 (A + B) * C라는 중위 표기법(Infix)을 쓸 때, 컴파일러는 이를 연산자 우선순위와 괄호에 구애받지 않도록 괄호가 없는 후위 표기법(Postfix) A B + C *로 변환한다. 이때 스택이 핵심적으로 사용되며, 웹 브라우저가 HTML 태그의 짝이 맞는지 검사할 때도 스택을 쓴다.
